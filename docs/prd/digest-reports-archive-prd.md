# Digest Reports 网页归档 PRD

> 文档类型：Traditional PRD
>
> 文档状态：Draft v1
>
> 更新时间：2026-04-09

## 1. 背景

当前系统已经会持久化 `digest_reports` 与 `digest_items`，后端生成的真正日报产物是 `digest_reports.markdownBody`。但前台现有 `/digest` 页面展示的仍是按评分排序的 `analysis_records` 列表，并不是已发布的历史日报归档。

这带来三个实际问题：

1. 用户错过邮件后，无法在网页上回看某一天已经生成的日报。
2. `digest_reports` 已经具备“历史快照”价值，但前端没有对应读取与展示入口。
3. `/digest` 现有语义更接近“编排候选/今日概览”，与“日报归档”容易混淆。

因此，本需求的目标不是替换现有 `/digest`，而是新增一套面向 `digest_reports` 的网页归档能力，并明确两类页面的产品语义。

## 2. 产品决策

### 2.1 页面语义拆分

- `/digest`：保留，继续承载“今日候选 / 编排候选 / 实时概览”语义。
- `/digests`：新增，承载“日报归档”列表。
- `/digests/[date]`：新增，承载“某一天日报详情”。

### 2.2 命名建议

为避免用户混淆，推荐调整导航文案而不是在本期强行改动旧路由：

- `/digest` 导航文案：`Digest 候选` 或 `今日候选`
- `/digests` 导航文案：`Digest 归档`

说明：

- 本期仍只支持 `daily` digest，因此详情路由使用 `/digests/[date]`。
- 若未来上线 weekly digest，再升级为 `/digests/daily/[date]` 与 `/digests/weekly/[date]`。

## 3. 目标与非目标

### 3.1 目标

1. 让用户可以在网页上查看历史 `digest_reports`。
2. 保证网页详情与邮件正文内容一致，以 `markdownBody` 作为展示真源。
3. 让 `/digest` 与历史归档并存时的语义清晰，不再混用“候选流”和“已发布报告”。
4. 为后续从内容详情页、统计页等位置回链历史 digest 建立稳定 URL。

### 3.2 非目标

1. 本期不做投递状态反馈展示。
2. 本期不做 `markdownBody` 全文检索。
3. 本期不基于 `digest_items + analysis_records` 重组详情页主内容。
4. 本期不引入新的鉴权体系，访问边界沿用当前应用现状。
5. 本期不覆盖 weekly digest。

## 4. 用户与核心场景

### 4.1 目标用户

- 需要回看历史日报的个人用户
- 需要核对系统某日输出内容的操作者

### 4.2 核心场景

1. 用户今天没有看邮件，晚上回到网页补看当天 digest。
2. 用户想回顾前几天某个主题的编排结果，按日期进入历史日报。
3. 用户从内容详情页看到某内容被编入过 digest，希望跳回对应日报查看原上下文。

## 5. 功能需求

## 功能：FR-001 历史归档列表页

**优先级：** P0
**描述：** 提供历史日报索引入口，展示已可阅读的 `daily` digest。

### 需求

| ID | 需求 | 备注 |
|----|------|------|
| REQ-001 | 系统必须提供 `/digests` 页面，用于按日期倒序列出历史日报。 | 仅展示 `period = daily` |
| REQ-002 | 列表默认仅展示可阅读的 digest，即状态为 `ready` 或 `sent` 的报告。 | `draft`、`failed` 不进入默认归档 |
| REQ-003 | 每个列表项至少展示 `digestDate` 与详情入口。 | 日期为主标识 |
| REQ-004 | 列表项可选展示轻量元数据，如条目数、创建时间或简短摘要片段。 | 有则更好，但不是阻塞项 |
| REQ-005 | 当无历史日报时，页面展示明确空状态。 | 不报错，不出现技术性提示 |

### 约束

- 日期展示必须遵循 digest 业务时区，而不是服务器本地时区。
- 列表页不引入全文搜索、日历视图、月份折叠等增强交互。

### 依赖

- 依赖于：`digest_reports`
- 被依赖：详情页入口、后续站内回链

## 功能：FR-002 日报详情页

**优先级：** P0
**描述：** 展示某一天已经生成的日报正文。

### 需求

| ID | 需求 | 备注 |
|----|------|------|
| REQ-006 | 系统必须提供 `/digests/[date]` 页面，通过 `digestDate` 加 `period=daily` 查询唯一报告。 | 例如 `/digests/2026-04-08` |
| REQ-007 | 详情页必须以 `digest_reports.markdownBody` 作为正文展示真源。 | 不重组主内容 |
| REQ-008 | 当 `markdownBody` 为空、报告不存在，或状态不允许阅读时，页面返回明确不可用状态。 | 推荐 `notFound` |
| REQ-009 | 详情页必须展示当前日报日期标题。 | 页面主标题 |
| REQ-010 | 详情页必须提供“前一天 / 后一天”导航。 | 基于已可阅读报告集合 |
| REQ-011 | Markdown 渲染必须保证基础安全与稳定性。 | 不直接注入原始 HTML |

### 约束

- 详情页主内容不基于 `digest_items` 二次拼装，避免同一份日报在邮件与网页出现内容漂移。
- 若 Markdown 渲染组件支持扩展，仅允许加入与当前仓库 UI 一致的轻量样式，不引入重交互块。

### 依赖

- 依赖于：`digest_reports.markdownBody`
- 被依赖：分享链接、内容详情回链

## 功能：FR-003 站内导航与语义校正

**优先级：** P0
**描述：** 避免现有 `/digest` 与新增历史归档在产品语义上混淆。

### 需求

| ID | 需求 | 备注 |
|----|------|------|
| REQ-012 | 系统必须保留现有 `/digest` 页面。 | 本期不下线 |
| REQ-013 | 系统必须为归档页提供独立导航入口。 | 推荐 `/digests` |
| REQ-014 | 系统应调整导航文案，区分“候选流”和“历史归档”。 | 路由可不变，文案应调整 |
| REQ-015 | 详情页 URL 应稳定可复制，可直接作为站内分享或回链地址。 | 不要求分享按钮 |

### 约束

- 本期优先控制认知成本，不做旧路由迁移与全站大规模重命名。

### 依赖

- 依赖于：侧边栏、页面标题、站内链接
- 被依赖：后续从内容详情、统计页等位置跳转历史 digest

## 功能：FR-004 内容详情页回链增强

**优先级：** P1
**描述：** 让已经存在的 digest 关联信息能跳到真实的日报详情页。

### 需求

| ID | 需求 | 备注 |
|----|------|------|
| REQ-016 | 内容详情页中的 digest relation 应可跳转到对应 `/digests/[date]`。 | 现有数据已具备 `digestDate` |
| REQ-017 | 若对应 digest 不可阅读，则不展示可点击链接。 | 避免坏链 |

### 约束

- 本需求不要求重做内容详情布局，只补齐跳转能力。

### 依赖

- 依赖于：`digest_items` 与 `digest_reports` 的关联查询

## 6. 数据与查询要求

### 6.1 数据真源

- 归档详情页正文真源：`digest_reports.markdownBody`
- 归档详情页主标识：`digest_reports.digestDate`
- 归档可读状态：`digest_reports.status in ('ready', 'sent')`

### 6.2 列表查询

列表页至少需要以下字段：

- `id`
- `digestDate`
- `status`
- `createdAt`
- `updatedAt`

可选聚合字段：

- `itemCount`：由 `digest_items` 聚合得到

排序规则：

- 按 `digestDate desc`

过滤规则：

- `period = 'daily'`
- `status in ('ready', 'sent')`

### 6.3 详情查询

详情页至少需要以下字段：

- `id`
- `digestDate`
- `markdownBody`
- `status`

查询规则：

- 使用 `period = 'daily' and digestDate = :date` 精确命中

前后篇导航规则：

- 在同样的可读集合中，查询 `digestDate` 前一个和后一个可读报告

### 6.4 数据边界

- `digest_items` 本期只用于聚合数量、补充回链和未来扩展，不用于重建详情主内容。
- `failed` 与 `draft` 报告不进入用户归档主路径。

## 7. 交互与展示要求

### 7.1 列表页

- 采用简洁的时间倒序列表，不引入复杂筛选控件。
- 每个列表项应让用户一眼看出“这是一份哪一天的日报”。
- 页面空状态应明确说明当前没有历史日报，不应出现调度或系统内部术语。

### 7.2 详情页

- 页面主标题以日期为核心。
- 正文采用 Markdown 渲染，保持阅读连续性。
- 前后篇导航应固定放在用户容易感知的位置。
- 页面不展示“邮件已发送至某邮箱”“投递时间”等信息。

### 7.3 文案

- `/digest` 推荐名称：`Digest 候选`
- `/digests` 推荐名称：`Digest 归档`
- `/digests/[date]` 页面标题推荐：`日报 2026-04-08`

## 8. 非功能需求

## 性能

| 指标 | 目标 | 说明 |
|------|------|------|
| 列表页服务端查询 | 常规数据量下 < 500ms | 不含数据库网络抖动 |
| 详情页服务端查询 | 常规数据量下 < 500ms | 单条命中 |
| 首屏策略 | 以 RSC / SSR 为主 | 避免为 Markdown 展示引入不必要客户端状态 |

## 安全

| 需求 | 实现方式 |
|------|----------|
| Markdown 渲染安全 | 禁止直接信任原始 HTML，使用受控 Markdown 渲染链 |
| 数据边界 | 只读取已持久化 digest，不改写历史报告 |

## 一致性

| 需求 | 实现方式 |
|------|----------|
| 邮件与网页一致 | 详情页正文直接使用 `markdownBody` |
| 时区一致 | 页面日期标签与业务 digest 时区保持一致 |

## 9. 技术约束

**前端：**

- Next.js App Router
- React Server Components 优先
- Tailwind CSS 4 + 现有 UI 组件

**后端：**

- Drizzle ORM + PostgreSQL
- 查询层优先复用现有服务/页面查询模式

**业务约束：**

- 历史 digest 属于加工层数据，不得回写覆盖原始内容层。
- 所有展示必须保持来源可追溯原则，但本期不要求把追溯信息从 Markdown 中再拆成独立 UI 模块。
- 数据库存储是 UTC，前端按配置时区展示日期。

## 10. 验收标准

### 功能：历史归档列表页

| ID | 标准 | 优先级 |
|----|------|--------|
| AC-1 | 用户访问 `/digests` 时，可以看到按日期倒序排列的 `daily` digest 列表。 | 关键 |
| AC-2 | `draft` 与 `failed` 状态的 digest 不出现在默认归档列表中。 | 关键 |
| AC-3 | 当数据库没有可读 digest 时，列表页展示清晰空状态。 | 高 |

### 功能：日报详情页

| ID | 标准 | 优先级 |
|----|------|--------|
| AC-4 | 用户访问 `/digests/2026-04-08` 时，系统能根据 `period + digestDate` 精确返回对应日报。 | 关键 |
| AC-5 | 详情页正文与数据库中的 `markdownBody` 一致，不通过 `digest_items` 重拼正文。 | 关键 |
| AC-6 | 用户可以从详情页切换到上一份或下一份可读日报。 | 高 |
| AC-7 | 对不存在、不可读或正文为空的 digest，页面返回明确不可用状态。 | 高 |

### 功能：站内导航

| ID | 标准 | 优先级 |
|----|------|--------|
| AC-8 | 侧边栏同时提供 `/digest` 与 `/digests` 的入口，且文案能区分两者语义。 | 关键 |
| AC-9 | 内容详情页中已存在的 digest relation 在完成 P1 后可跳到对应历史日报详情。 | 中 |

## 11. 实施切分建议

### Phase 1：归档可读闭环

1. 新增 `digests` 查询层：列表、详情、前后篇导航。
2. 新增 `/digests` 与 `/digests/[date]` 页面。
3. 接入 Markdown 渲染与基础样式。
4. 调整侧边栏文案与入口，区分 `/digest` 和 `/digests`。

### Phase 2：站内回链增强

1. 在内容详情页的 digest relation 区块中补充跳转链接。
2. 视页面信息密度决定是否补充条目数等轻量元数据。

## 12. 已决策事项

1. `/digest` 页面保留，不在本期替换或下线。
2. 历史归档详情以 `markdownBody` 为真源。
3. 本期不做投递状态反馈。
4. 本期不做全文检索。
5. 本期不做高度可交互仪表盘式重构，优先交付稳定归档能力。
