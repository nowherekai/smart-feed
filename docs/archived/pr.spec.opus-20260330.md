## 1. 产品环境 (Product Environment)

### 1.1 产品定位

**名称**: smart-feed

**标语**: 将自定义订阅源转化为每日可追溯的个人情报摘要

**环境描述**:
smart-feed 是一个为知识工作者设计的信息操作环境。它提供了一个行动空间，让用户能够配置信息源、让系统自动抓取和分析内容、生成个性化摘要报告，并通过反馈持续优化信息流。这个环境强调可追溯性（每条结论都能回溯到原文）和成本控制（分层处理避免 AI 成本失控）。

### 1.2 主要行动者 (Primary Agents)

**人类用户 (Human Agent)**:
- 行动能力: 配置信息源、阅读摘要报告、提供反馈（有用/没用/屏蔽）、调整偏好
- 感知通道: 视觉界面、邮件通知
- 目标: 高效获取个性化信息情报，减少信息过载

**AI 分析器 (AI Agent)**:
- 行动能力: 抓取内容、清洗数据、分类、提取关键词、生成摘要、编排报告
- 感知通道: RSS Feed、API、结构化数据
- 目标: 准确理解内容、生成高质量摘要、优化成本

### 1.3 核心可供性 (Core Affordances)

1. **配置信息源** - 用户能够添加、管理和控制信息来源
2. **自动内容获取** - 系统能够定时抓取和标准化内容
3. **智能内容分析** - AI 能够分类、摘要和评估内容价值
4. **生成摘要报告** - 系统能够编排和投递个性化 digest
5. **反馈优化** - 用户能够通过反馈影响后续推荐

## 2. 最小可供性故事 (Minimum Affordance Stories)

### MAS-1: 订阅并获取信息流

**故事主题**:
用户配置感兴趣的 RSS 订阅源，系统自动抓取内容并标准化存储

**核心可供性序列**:
1. **添加信息源** (perceive: 添加按钮/导入功能)
   - 单个添加 RSS 订阅源
   - 批量导入 OPML 文件
   - 系统验证源的有效性

2. **管理信息源** (perceive: 信息源列表)
   - 查看已配置的信息源
   - 启用/暂停/删除信息源
   - 调整信息源权重

3. **自动抓取内容** (perceive: 后台定时任务)
   - 系统定时抓取 RSS 内容
   - 清洗 HTML 转为 Markdown
   - 去重并标准化存储

**意义闭环**:
用户不再需要手动访问多个网站，信息自动汇聚到统一的内容池中，为后续分析做好准备。

**内在动机**:
- 兴趣: 掌控自己的信息源
- 精通: 逐步优化订阅列表
- 自主: 完全自定义信息来源

**故事依赖**:
- 启用: MAS-2 (分析与摘要)
- 需要: 无

**不包含的内容**:
- AI 分析（属于 MAS-2）
- 报告生成（属于 MAS-3）
- 反馈优化（属于 MAS-4）

---

### MAS-2: 智能分析与摘要

**故事主题**:
AI 自动分析内容价值、分类并生成摘要，节省用户阅读时间

**核心可供性序列**:
1. **基础内容分析** (perceive: 分析记录)
   - 主题分类
   - 关键词提取
   - 实体识别（公司、人名、产品）
   - 价值评分（是否值得读）

2. **单篇摘要生成** (perceive: 摘要字段)
   - 一句话总结
   - 三点核心摘要
   - "为什么值得关注"说明

3. **可追溯性保证** (perceive: 原文链接)
   - 每条结论关联原文片段
   - 保留原始链接和来源信息
   - 支持一键跳转到原文

**意义闭环**:
用户能够快速了解内容核心，决定是否深入阅读，同时保持对信息来源的信任。

**内在动机**:
- 兴趣: 发现高价值内容
- 精通: 理解 AI 如何筛选信息
- 自主: 可以验证 AI 的判断

**故事依赖**:
- 启用: MAS-3 (报告编排)
- 需要: MAS-1 (内容已抓取)

**不包含的内容**:
- 报告编排（属于 MAS-3）
- 用户反馈（属于 MAS-4）

---

### MAS-3: 生成并投递摘要报告

**故事主题**:
系统每日自动编排 digest 报告并投递给用户

**核心可供性序列**:
1. **报告编排** (perceive: digest 文档)
   - 按主题分类组织内容
   - 按价值排序
   - 复用已缓存的摘要结果

2. **报告生成** (perceive: 日报/周报文件)
   - 生成 Markdown 格式报告
   - 包含标题、摘要、原文链接
   - 标注来源和发布时间

3. **报告投递** (perceive: 邮件通知)
   - 每日定时发送邮件
   - 或存放到指定目录并通知用户
   - 支持归档查看历史报告

**意义闭环**:
用户每天收到个性化的信息摘要，无需主动查找，信息主动送达。

**内在动机**:
- 兴趣: 期待每日新发现
- 精通: 建立稳定的信息消费习惯
- 自主: 控制投递时间和方式

**故事依赖**:
- 启用: MAS-4 (反馈优化)
- 需要: MAS-2 (内容已分析)

**不包含的内容**:
- 用户反馈（属于 MAS-4）
- 来源权重调整（属于 MAS-4）

---

### MAS-4: 反馈闭环优化

**故事主题**:
用户通过反馈信号持续优化信息流，让推荐越来越精准

**核心可供性序列**:
1. **内容反馈** (perceive: 反馈按钮)
   - 标记"有用/没用"
   - 屏蔽特定来源
   - 降低某类主题权重

2. **偏好调整** (perceive: 设置界面)
   - 选择摘要风格（短摘要/深分析/行动导向）
   - 调整来源权重
   - 设置主题偏好

3. **系统优化** (perceive: 推荐改进)
   - 根据反馈调整后续排序
   - 减少低价值内容推送
   - 增加高价值来源权重

**意义闭环**:
系统不再"自说自话"，而是根据用户真实需求持续进化，信息流越来越贴合个人需求。

**内在动机**:
- 兴趣: 看到系统在学习自己的偏好
- 精通: 掌握如何训练系统
- 自主: 完全控制信息流方向

**故事依赖**:
- 启用: 无（已完成核心闭环）
- 需要: MAS-3 (报告已投递)

**不包含的内容**:
- 多来源扩展（属于未来版本）

## 3. 环境约束 (Environmental Constraints)

### 3.1 物理约束

| 约束 | 对可供性的影响 |
|------|---------------|
| 网络带宽 | 限制同时抓取的信息源数量 |
| 存储空间 | 需要定期清理旧内容或归档 |
| AI API 限流 | 需要分层处理和缓存策略 |

### 3.2 结构约束

| 约束 | 对可供性的影响 |
|------|---------------|
| 数据模型 | 内容必须标准化为统一的 content-item 结构 |
| 来源类型 | MVP 仅支持 RSS，其他来源需扩展架构 |
| 分析流程 | 必须先抓取、再清洗、再分析，不可跳步 |

### 3.3 安全约束 (来自 real.md)

| 约束 | 可供性限制 |
|------|-----------|
| 授权访问 | 只能采集用户明确配置的信息源 |
| 数据分离 | 原始数据与加工数据必须分开存储 |
| 可追溯性 | 所有 AI 结论必须可回溯到原文 |
| 反馈响应 | 用户屏蔽/降权后必须立即生效 |
| 成本控制 | 必须分层处理，避免全文重模型分析 |
| 隐私保护 | 不存储音视频文件，仅处理文本转写稿 |

## 4. 感知通道 (Perception Channels)

### 4.1 人类用户感知

| 元素 | 感知到的行动 | 设计模式 |
|------|-------------|---------|
| 添加按钮 | 可添加信息源 | 明显的 "+" 或 "添加" 按钮 |
| 信息源列表 | 可查看和管理 | 列表视图，带状态标识 |
| 摘要卡片 | 可阅读摘要 | 卡片式布局，标题+摘要+链接 |
| 反馈按钮 | 可标记有用/没用 | 👍/👎 图标或文字按钮 |
| 邮件通知 | 有新报告 | 邮件主题明确标注日期 |

### 4.2 AI 代理感知

| 元素 | 语义标记 | 感知到的行动 |
|------|---------|-------------|
| RSS Feed | XML 结构 | 可解析的内容项 |
| content-item | 数据库记录 | 可分析的标准化内容 |
| analysis-record | 关联字段 | 可复用的分析结果 |
| feedback-signal | 事件记录 | 可学习的用户偏好 |

### 4.3 跨模态可供性

| 可供性 | 人类通道 | AI 通道 | 统一行动 |
|--------|---------|---------|---------|
| 添加信息源 | 点击按钮输入 URL | API 调用 | 信息源入库 |
| 查看摘要 | 阅读邮件/界面 | 读取 digest 记录 | 获取摘要内容 |
| 提供反馈 | 点击反馈按钮 | 写入 feedback-signal | 更新偏好模型 |

## 5. 反馈机制 (Feedback Mechanisms)

### 5.1 即时反馈 (< 100ms)

| 行动 | 人类反馈 | AI 反馈 |
|------|---------|---------|
| 点击添加 | 按钮状态变化 | DOM 事件触发 |
| 输入 URL | 字符实时显示 | 输入值变化事件 |
| 点击反馈按钮 | 图标状态改变 | 状态更新事件 |

### 5.2 进行中反馈 (< 1s)

| 行动 | 人类反馈 | AI 反馈 |
|------|---------|---------|
| 添加信息源 | "验证中..." 提示 | 验证状态 |
| 抓取内容 | 进度指示器 | 抓取进度事件 |
| 生成摘要 | "分析中..." 提示 | 处理状态 |

### 5.3 完成反馈 (< 5s)

| 行动 | 人类反馈 | AI 反馈 |
|------|---------|---------|
| 添加信息源 | "添加成功" 消息 | 信息源记录已创建 |
| 抓取内容 | "已抓取 N 条内容" | 内容记录已入库 |
| 生成报告 | "报告已发送" | digest 记录已生成 |
| 提供反馈 | "已记录您的反馈" | feedback-signal 已保存 |

## 6. 详细可供性规格 (Detailed Affordance Specifications)

### 6.1 MAS-1 相关可供性

#### AFF-1.1: 添加单个 RSS 源

**级别**: 主要可供性

**启用的行动**: 用户输入 RSS URL 并添加到系统

**环境属性**:

| 属性 | 值 | 可被感知 |
|------|---|---------|
| 视觉提示 | "添加信息源" 按钮 | 人类 |
| 输入框 | URL 输入字段 | 人类 |
| API 端点 | POST /api/sources | AI 代理 |

**代理要求**:

| 代理类型 | 所需能力 |
|---------|---------|
| 人类 | 视觉、输入设备、理解 RSS 概念 |
| AI 代理 | HTTP 请求能力、RSS 解析能力 |

**感知-行动耦合**:

**人类感知**:
- 看到: "添加信息源" 按钮
- 做: 点击按钮，输入 URL，确认
- 反馈: "添加成功" 或 "URL 无效"

**AI 感知**:
- 检测: POST /api/sources 端点
- 执行: 发送 JSON {url: "..."}
- 验证: 返回 201 状态码和 source_id

**约束**:
- URL 必须是有效的 RSS/Atom feed
- 不允许重复添加相同 URL（提示而非报错）
- 需要网络连接验证源可访问性

**依赖**:
- 需要: 无
- 启用: AFF-1.2 (管理信息源)

---

#### AFF-1.2: 批量导入 OPML

**级别**: 主要可供性

**启用的行动**: 用户上传 OPML 文件批量导入订阅源

**环境属性**:

| 属性 | 值 | 可被感知 |
|------|---|---------|
| 视觉提示 | "导入 OPML" 按钮 | 人类 |
| 文件选择器 | 文件上传区域 | 人类 |
| API 端点 | POST /api/sources/import | AI 代理 |

**感知-行动耦合**:

**人类感知**:
- 看到: "导入 OPML" 按钮
- 做: 选择 OPML 文件，上传
- 反馈: "成功导入 N 个源，M 个已存在"

**AI 感知**:
- 检测: multipart/form-data 上传端点
- 执行: 上传 OPML 文件
- 验证: 返回导入结果统计

**约束**:
- 文件必须是有效的 OPML 格式
- 已存在的源不报错，仅提示
- 单次导入限制 1000 个源

**依赖**:
- 需要: 无
- 启用: AFF-1.2 (管理信息源)

---

#### AFF-1.3: 管理信息源状态

**级别**: 主要可供性

**启用的行动**: 用户查看、启用、暂停、删除信息源

**环境属性**:

| 属性 | 值 | 可被感知 |
|------|---|---------|
| 视觉提示 | 信息源列表，带状态标识 | 人类 |
| 操作按钮 | 启用/暂停/删除按钮 | 人类 |
| API 端点 | PATCH /api/sources/:id | AI 代理 |

**感知-行动耦合**:

**人类感知**:
- 看到: 信息源列表，每项显示名称、URL、状态
- 做: 点击操作按钮改变状态
- 反馈: 状态即时更新，"已暂停" 或 "已删除"

**AI 感知**:
- 检测: PATCH /api/sources/:id 端点
- 执行: 发送 {status: "active|paused|blocked"}
- 验证: 返回更新后的 source 对象

**约束**:
- 暂停的源不会被抓取
- 删除操作需要二次确认
- 删除源不会删除已抓取的内容

**依赖**:
- 需要: AFF-1.1 或 AFF-1.2 (已有信息源)
- 启用: AFF-1.4 (自动抓取)

### 6.2 MAS-2 相关可供性

#### AFF-2.1: 基础内容分析

**级别**: 主要可供性

**启用的行动**: AI 自动分析内容并生成结构化结果

**环境属性**:

| 属性 | 值 | 可被感知 |
|------|---|---------|
| 输入 | content-item 记录 | AI 代理 |
| 输出 | analysis-record 记录 | AI 代理 |
| 分析维度 | 主题、关键词、实体、价值评分 | 系统 |

**感知-行动耦合**:

**AI 感知**:
- 检测: 新的 content-item 记录（未分析）
- 执行: 调用轻量级模型进行分类和抽取
- 验证: 生成 analysis-record 并关联

**约束**:
- 优先使用轻模型（成本控制）
- 结果必须可缓存和复用
- 分析失败不阻塞后续流程

**依赖**:
- 需要: content-item 已入库
- 启用: AFF-2.2 (单篇摘要)

---

#### AFF-2.2: 单篇摘要生成

**级别**: 主要可供性

**启用的行动**: AI 为高价值内容生成摘要

**环境属性**:

| 属性 | 值 | 可被感知 |
|------|---|---------|
| 输入 | 高价值 content-item | AI 代理 |
| 输出 | 摘要字段（一句话、三点、理由） | AI 代理 |
| 策略 | 仅对价值评分 > 阈值的内容生成 | 系统 |

**感知-行动耦合**:

**AI 感知**:
- 检测: analysis-record 中价值评分 > 阈值
- 执行: 调用重模型生成摘要
- 验证: 摘要已保存到 analysis-record

**约束**:
- 仅对高价值内容生成摘要（成本控制）
- 摘要必须包含原文引用
- 结果必须缓存，避免重复生成

**依赖**:
- 需要: AFF-2.1 (已完成基础分析)
- 启用: AFF-3.1 (报告编排)

---

### 6.3 MAS-3 相关可供性

#### AFF-3.1: 报告编排

**级别**: 主要可供性

**启用的行动**: 系统按主题和价值编排 digest

**环境属性**:

| 属性 | 值 | 可被感知 |
|------|---|---------|
| 输入 | 时间范围内的 analysis-record | 系统 |
| 输出 | digest-report 记录 | 系统 |
| 编排规则 | 按主题分组、按价值排序 | 系统 |

**感知-行动耦合**:

**系统感知**:
- 检测: 定时触发（每日固定时间）
- 执行: 查询未编排的内容，按规则组织
- 验证: digest-report 已生成

**约束**:
- 复用已缓存的摘要结果
- 低价值内容可选择性排除
- 同一内容不重复出现在同一报告

**依赖**:
- 需要: AFF-2.2 (已有摘要)
- 启用: AFF-3.2 (报告投递)

---

#### AFF-3.2: 报告投递

**级别**: 主要可供性

**启用的行动**: 系统将报告发送给用户

**环境属性**:

| 属性 | 值 | 可被感知 |
|------|---|---------|
| 投递方式 | 邮件或文件通知 | 人类 |
| 报告格式 | Markdown | 人类/AI |
| 投递时间 | 用户配置的时间 | 系统 |

**感知-行动耦合**:

**人类感知**:
- 看到: 收到邮件或通知
- 做: 打开邮件或文件阅读
- 反馈: 可点击链接跳转原文

**约束**:
- 邮件主题包含日期和内容数量
- 报告包含原文链接和来源信息
- 支持归档查看历史报告

**依赖**:
- 需要: AFF-3.1 (报告已编排)
- 启用: AFF-4.1 (用户反馈)

---

### 6.4 MAS-4 相关可供性

#### AFF-4.1: 内容反馈

**级别**: 主要可供性

**启用的行动**: 用户标记内容有用/没用

**环境属性**:

| 属性 | 值 | 可被感知 |
|------|---|---------|
| 视觉提示 | 👍/👎 按钮 | 人类 |
| API 端点 | POST /api/feedback | AI 代理 |

**感知-行动耦合**:

**人类感知**:
- 看到: 每条摘要下方的反馈按钮
- 做: 点击 👍 或 👎
- 反馈: "已记录您的反馈"

**AI 感知**:
- 检测: POST /api/feedback 端点
- 执行: 发送 {content_id, signal: "useful|useless"}
- 验证: feedback-signal 已保存

**约束**:
- 反馈立即生效，影响后续推荐
- 支持撤销反馈
- 反馈数据用于优化排序和来源权重

**依赖**:
- 需要: AFF-3.2 (报告已投递)
- 启用: 系统持续优化

#### AFF-4.2: 来源权重调整

**级别**: 次要可供性

**启用的行动**: 用户调整或屏蔽信息源

**环境属性**:

| 属性 | 值 | 可被感知 |
|------|---|---------|
| 视觉提示 | 来源管理界面 | 人类 |
| 操作 | 权重滑块、屏蔽按钮 | 人类 |
| API 端点 | PATCH /api/sources/:id/weight | AI 代理 |

**感知-行动耦合**:

**人类感知**:
- 看到: 来源列表，带权重控制
- 做: 调整权重或点击屏蔽
- 反馈: "权重已更新" 或 "已屏蔽该来源"

**约束**:
- 屏蔽后该源内容不再出现在报告中
- 权重影响内容在报告中的排序
- 支持临时屏蔽和永久屏蔽

**依赖**:
- 需要: AFF-1.3 (已有信息源)
- 启用: 系统持续优化

## 8. 非可供性 (Non-Affordances)

## 9. 技术实现 (Technical Implementation)

### 9.1 数据模型

基于 cog.md 中定义的实体模型：

**source (信息源)**
```typescript
{
  id: UUID
  type: "rss" | "podcast" | "newsletter" | "wechat" | "youtube"
  identifier: string  // 规范化标识，如 URL
  name: string
  status: "active" | "paused" | "blocked"
  weight: number  // 权重，影响排序
  created_at: timestamp
  updated_at: timestamp
}
```

**content-item (内容项)**
```typescript
{
  id: UUID
  source_id: UUID  // 关联 source
  external_id: string  // 来源侧唯一 ID
  title: string
  author: string
  original_url: string
  raw_body: string  // 原始 HTML
  cleaned_md: string  // 清洗后的 Markdown
  media_url: string  // 封面图
  published_at: timestamp
  fetched_at: timestamp
}
```

**analysis-record (分析记录)**
```typescript
{
  id: UUID
  content_id: UUID  // 关联 content-item
  model_strategy: string  // 使用的模型策略
  version: string
  category: string[]  // 主题分类
  keywords: string[]
  entities: object  // {companies, persons, products}
  value_score: number  // 价值评分
  summary_oneline: string
  summary_points: string[]
  summary_reason: string
  created_at: timestamp
}
```

**digest-report (摘要报告)**
```typescript
{
  id: UUID
  period: "daily" | "weekly"
  date: string  // YYYY-MM-DD
  content_ids: UUID[]  // 包含的内容
  report_md: string  // 报告 Markdown
  delivered_at: timestamp
  created_at: timestamp
}
```

**feedback-signal (反馈信号)**
```typescript
{
  id: UUID
  target_type: "content" | "source" | "topic"
  target_id: UUID
  signal: "useful" | "useless" | "block" | "increase_weight" | "decrease_weight"
  user_id: UUID
  created_at: timestamp
}
```

### 9.2 核心流程

**流程 1: 信息源抓取**
```
定时触发 → 查询 active 状态的 source 
→ 并发抓取 RSS feed 
→ 解析 XML 提取内容 
→ 清洗 HTML 转 Markdown 
→ 去重检查 (external_id) 
→ 入库 content-item
```

**流程 2: 内容分析**
```
检测未分析的 content-item 
→ 轻模型分类和抽取 (主题、关键词、实体、价值评分) 
→ 保存 analysis-record 
→ 如果 value_score > 阈值 
  → 重模型生成摘要 
  → 更新 analysis-record
```

**流程 3: 报告生成**
```
定时触发 (每日固定时间) 
→ 查询时间范围内的 analysis-record 
→ 按主题分组 
→ 按价值排序 
→ 复用缓存的摘要 
→ 编排 Markdown 报告 
→ 保存 digest-report 
→ 投递邮件或通知
```

**流程 4: 反馈处理**
```
用户提交反馈 
→ 保存 feedback-signal 
→ 更新相关权重 
  - 内容反馈 → 影响后续排序 
  - 来源反馈 → 更新 source.weight 或 status 
  - 主题反馈 → 更新用户偏好模型
```

### 9.3 成本控制策略

| 策略 | 实现方式 |
|------|---------|
| 分层处理 | 轻模型做分类，重模型做摘要 |
| 选择性摘要 | 仅对 value_score > 阈值的内容生成摘要 |
| 结果缓存 | analysis-record 永久缓存，避免重复分析 |
| 内容裁剪 | 超长内容截断或分段处理 |
| 批量处理 | 批量调用 API 降低请求次数 |

## 10. 附录 (Appendix)

### 10.1 可供性理论参考

**核心概念**:
- **可供性 (Affordance)**: 环境属性与代理行动能力之间的关系
- **感知-行动耦合**: 代理感知环境线索并执行相应行动
- **反馈闭环**: 行动结果反馈给代理，形成学习循环

**设计原则**:
1. 可供性必须可感知（人类通过视觉，AI 通过语义）
2. 行动必须可执行（匹配代理能力）
3. 反馈必须及时清晰（确认行动结果）
4. 约束必须明确（防止无效行动）

### 10.2 术语表

| 术语 | 定义 |
|------|------|
| MAS | Minimum Affordance Story，最小可供性故事 |
| 可供性 | 环境提供给代理的行动可能性 |
| 代理 | 能够感知和行动的实体（人类或 AI） |
| 感知通道 | 代理获取环境信息的方式 |
| 反馈机制 | 环境向代理确认行动结果的方式 |
| 环境约束 | 限制可供性的物理、结构或安全因素 |
| content-item | 标准化的内容项，系统的基础数据单位 |
| analysis-record | 针对内容的结构化分析结果 |
| digest-report | 编排后的摘要报告 |
| feedback-signal | 用户反馈信号，用于优化系统 |
