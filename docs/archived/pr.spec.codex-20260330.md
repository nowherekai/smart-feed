## 1. 文档定位

本文件用于定义 `smart-feed` 的产品环境、核心 affordance、Minimum Affordance Story（MAS）、约束条件、最小产品接口与验收标准，供产品、工程与 AI agent 协同实现时统一理解与边界。

本文档采用 affordance-driven 结构，不采用传统功能清单式 PRD。目标不是罗列“做哪些功能”，而是明确这个产品环境为不同 agent 提供了哪些可感知、可执行、可验证的行动可能性。

## 3. 产品环境

### 3.1 产品环境定义

**Name:** smart-feed

**Tagline:** 将自定义订阅源转化为每日可追溯的个人情报摘要

**Version:** 0.1.0

**Environmental Description:**

`smart-feed` 是一个个人信息情报环境。它允许用户把自己明确选择的信息源接入系统，由系统完成信息获取、清洗标准化、AI 分析和 digest 编排，再将结果以日报形式投递给用户。

这个环境的核心不是“替用户阅读一切”，而是为用户提供一个可控、可追溯、可调优的信息处理空间：用户决定来源，系统完成结构化分析，AI 输出必须能回到原文证据，用户反馈必须实质性改变下一轮输出。

### 3.2 主要 Agent

| Agent | 类型 | 主要目标 | 感知方式 | 可执行动作 |
| --- | --- | --- | --- | --- |
| 人类用户 | Human | 配置来源、消费 digest、建立对结论的信任、持续调优输出 | 来源配置入口、digest 邮件/报告、原文回链、证据片段、反馈入口 | 添加来源、批量导入、查看 digest、打开原文、提交反馈、调整偏好 |
| AI 分析代理 | AI | 对内容做分类、抽取、摘要、编排，同时满足可追溯与成本约束 | 标准化内容模型、来源元数据、分析缓存、反馈信号、约束规则 | 分类、关键词抽取、实体识别、价值判断、单篇摘要、候选内容筛选、digest 编排 |
| 系统调度代理 | System | 在正确时间触发抓取、分析、投递与反馈生效 | 来源状态、抓取计划、内容状态、digest 周期、投递目标 | 定时抓取、触发流水线、生成日报、发出投递、应用反馈后的权重与过滤规则 |

### 3.3 核心 Affordance

1. 来源接入 affordance：允许用户显式添加或批量导入 RSS 来源。
2. 内容转化 affordance：允许系统把原始内容转化为统一内部内容模型。
3. 分析编排 affordance：允许 AI 在分层处理前提下生成单篇摘要与 digest。
4. 可追溯阅读 affordance：允许用户从 digest 结论回到来源、原文链接与证据片段。
5. 调优反馈 affordance：允许用户用明确反馈改变后续排序、曝光与摘要风格。

## 4. MAS：Minimum Affordance Stories

### 4.1 MAS-1：从 RSS 到每日可追溯 Digest

**Story Theme：**
把用户自选的 RSS 来源转化为一份每天都可回溯原文的个人情报摘要。

**Primary Agents：**
人类用户、系统调度代理、AI 分析代理。

**触发条件：**
用户至少配置了一个 `active` 状态的 RSS 来源，系统到达抓取与日报生成时间。

**Core Affordance Sequence：**

1. 入口 affordance：用户感知到来源配置入口并添加单条 RSS，或通过批量导入接入已有 RSS 清单。
2. 去重 affordance：系统在导入时识别已存在来源，不报错，而是提示“已存在/已跳过/已新增”的结果。
3. 抓取 affordance：系统按计划读取 `active` RSS 来源，获取新增内容并保存原始数据。
4. 标准化 affordance：系统将原始 RSS 内容清洗、转换为 Markdown、去重并写入统一 `content-item`。
5. 轻分析 affordance：AI 先完成主题分类、关键词、实体、语言、情绪、是否值得读等基础抽取。
6. 摘要 affordance：AI 对候选内容生成一句话总结、三点摘要与“为什么值得关注”。
7. 编排 affordance：系统按主题组织内容，形成 `daily` digest。
8. 可追溯 affordance：digest 中每条结论都附带来源、原始链接与证据片段。
9. 投递 affordance：系统默认以电子邮件投递日报，并保留报告形态以供查看或归档。

**Perceive → Act → Feedback 闭环：**

- 用户感知：看到来源入口、日报内容、主题结构、证据回链。
- 用户行动：导入来源、阅读 digest、点击原文。
- 系统反馈：返回导入结果、生成日报、提供原文证据链，建立对 AI 结论的信任。

**Meaning Closure：**

用户不再需要逐个源头查看当天所有信息，而是在自己选择的来源集合上获得一份可信、节省时间、可回看原文依据的摘要。

**Intrinsic Motivation：**

- Interest：每日从自己关心的来源中发现值得看的内容。
- Mastery：逐步理解哪些来源更有价值、哪些主题更值得保留。
- Autonomy：来源由用户自己决定，是否深读由用户自己决定。

**Story Dependencies：**

- Requires：无
- Enables：MAS-2

**Out of Scope：**

- 非 RSS 来源的 MVP 交付
- 未经用户授权的自动抓取
- 无出处的黑盒摘要
- 复杂 Web 工作台或重型可视化后台

### 4.2 MAS-2：用反馈重塑后续输出

**Story Theme：**
让 digest 不再是单向输出，而是可以被用户反馈持续校正的个人情报流。

**Primary Agents：**
人类用户、系统调度代理、AI 分析代理。

**触发条件：**
用户已经收到至少一份 digest，并对某条内容、某个来源、某类主题或摘要风格有明确判断。

**Core Affordance Sequence：**

1. 反馈入口 affordance：用户在 digest 项目、来源级别或偏好设置中感知到反馈入口。
2. 内容反馈 affordance：用户对单条内容标记“有用 / 没用”。
3. 来源控制 affordance：用户对来源执行“屏蔽来源”或后续可扩展的权重调节。
4. 偏好控制 affordance：用户降低某类主题偏好，或声明更喜欢“短摘要 / 深分析 / 行动导向”。
5. 规则生效 affordance：系统将 `feedback-signal` 持久化，并更新下一轮候选排序、过滤和摘要风格策略。
6. 输出变化 affordance：下一次 digest 中，用户能观察到反馈已经改变曝光、排序或摘要呈现。

**Perceive → Act → Feedback 闭环：**

- 用户感知：digest 中可操作的反馈入口与来源控制入口。
- 用户行动：点“没用”、屏蔽来源、降低主题权重、表达摘要偏好。
- 系统反馈：即时记录反馈，并在下一轮 digest 中体现可观察变化。

**Meaning Closure：**

系统从“AI 自己总结给你看”转变为“可被你驯化的个人情报助手”，用户通过反馈不断缩小噪声、强化有价值来源与表达形式。

**Intrinsic Motivation：**

- Interest：看到输出越来越贴合自己的判断。
- Mastery：逐渐形成稳定的个人信息过滤系统。
- Autonomy：用户拥有明确控制权，而不是被模型隐式决定。

**Story Dependencies：**

- Requires：MAS-1
- Enables：更长期的个性化与来源扩展能力

**Out of Scope：**

- 仅凭隐式行为全自动推断偏好而不记录显式反馈
- 在未经过用户控制的情况下恢复被屏蔽来源
- 把反馈当作纯展示性动作而不改变后续编排

### 4.3 MAS-3：扩展到更多来源

**Story Theme：**
在不破坏统一内容模型、可追溯性和成本控制原则的前提下，把 RSS 之外的文本化来源纳入同一情报环境。

**Primary Agents：**
人类用户、系统调度代理、AI 分析代理。

**触发条件：**
MAS-1 与 MAS-2 稳定，且产品决定扩展来源类型。

**Core Affordance Sequence：**

1. 来源扩展 affordance：用户感知到除 RSS 外的来源入口，例如 Podcast、Newsletter、微信公众号。
2. 文本前提 affordance：系统只接入文本原文、字幕、转写稿或等价文本结果，不接收原始音视频文件。
3. 统一建模 affordance：新增来源仍必须落到相同 `source`、`content-item`、`analysis-record`、`digest-report`、`feedback-signal` 关系中。
4. 复用流水线 affordance：新增来源复用现有清洗、分析、编排和反馈闭环。

**Perceive → Act → Feedback 闭环：**

- 用户感知：系统开始支持新来源类型。
- 用户行动：添加新增来源。
- 系统反馈：新增来源内容能进入同一摘要、回链与反馈系统。

**Meaning Closure：**

用户获得一个跨来源但逻辑一致的个人情报系统，而不是多个彼此割裂的阅读工具。

**Story Dependencies：**

- Requires：MAS-1、MAS-2
- Enables：更广泛的信息覆盖与统一管理

**Out of Scope：**

- 作为当前 MVP 的必须交付
- 存储和处理音频/视频原文件
- 为不同来源建立完全不同的一套分析和反馈协议

## 6. 领域对象与最小产品接口

### 6.1 规范术语

本文档固定使用以下领域对象作为产品与工程共用术语：

- `source`
- `content-item`
- `analysis-record`
- `digest-report`
- `feedback-signal`

这些对象的基本关系沿用 `.42cog/cog.md` 中的定义，不在本文档中重新发明新的核心实体。

### 6.2 领域对象要求

| 对象 | 作用 | 关键字段 / 状态 | 产品层要求 |
| --- | --- | --- | --- |
| `source` | 信息源入口 | `id`、`type`、规范化标识、状态 | 至少支持 `active / paused / blocked`；MVP 只要求 `rss-source`；批量导入时按规范化标识去重 |
| `content-item` | 统一后的内容单元 | `source_id`、`external_id`、`title`、`author`、`original_url`、`raw_body`、`cleaned_md`、`published_at`、`fetched_at` | 原始数据与加工数据分离；支持原始、标准化、去重等生命周期状态 |
| `analysis-record` | 单条内容的分析结果 | `content-item-id + model-strategy + version`、分类、关键词、实体、摘要、价值判断 | 必须支持版本化与缓存；区分基础抽取、单篇摘要、深度分析；重分析结果不可覆盖原始文本 |
| `digest-report` | 日报或周报载体 | `period + date`、内容分组、投递形态 | MVP 强制支持 `daily`；`weekly` 为后续扩展；默认邮件投递，同时保留报告形态用于查看或归档 |
| `feedback-signal` | 用户反馈控制信号 | `target-type + target-id + timestamp`、反馈动作 | 反馈必须影响后续编排；至少支持 `有用 / 没用 / 屏蔽来源 / 降低主题权重 / 提高来源权重 / 偏好短摘要 / 偏好深分析 / 偏好行动导向` |

### 6.3 Traceability 最小要求

所有面向用户的 AI 结论必须至少能够追溯到以下信息：

- `source_id`
- 来源名称或来源标识
- `original_url`
- `content-item` 标识
- 对应的证据片段或原文片段
- 产生该结论的 `analysis-record` 标识或等价版本信息

没有这些信息的结论不得进入最终 digest。

### 6.4 最小产品接口

#### 来源输入接口

系统至少需要支持以下输入动作，不限定入口形态是命令行、配置文件、后续表单还是其他产品入口：

1. 添加单条 RSS 来源
2. 批量导入 RSS 来源
3. 对重复来源做去重并提示“已存在/已跳过/已新增”
4. 调整来源状态为 `active / paused / blocked`

#### 内容输出接口

系统至少需要支持以下输出能力：

1. 生成 `daily` digest
2. 默认通过电子邮件投递
3. 每条 digest 内容至少包含：
   - 标题或主题归属
   - 一句话总结
   - 三点摘要或等价要点
   - 为什么值得关注
   - 来源
   - 原始链接
   - 证据片段

#### 反馈接口

系统至少需要支持以下反馈动作：

1. 对单条内容标记“有用 / 没用”
2. 对来源执行“屏蔽来源”
3. 降低某类主题偏好
4. 表达摘要风格偏好：短摘要 / 深分析 / 行动导向

这些反馈必须被记录为结构化 `feedback-signal`，并在下一轮编排或摘要生成中生效。

## 7. 环境约束

### 7.1 现实约束映射

| 现实约束 | 产品要求 |
| --- | --- |
| 只允许采集和处理用户明确配置或授权的信息源 | 只能处理用户添加的来源；不得自动扩展抓取范围；不得绕过来源平台公开访问边界 |
| 不存储也不直接处理音频或视频文件 | 后续音视频来源只接受字幕、转写稿或其他纯文本结果 |
| 原始数据与加工数据必须分离 | `content-item` 原文与 `analysis-record`、`digest-report` 结果分开管理，禁止覆盖原文 |
| 所有 AI 输出必须可回溯到来源、链接和片段 | digest 中每条结论必须携带 traceability 元数据 |
| 用户反馈必须被当作真实控制信号处理 | 反馈不是装饰性功能；必须改变后续排序、过滤或摘要风格 |
| AI 链路必须优先分层处理、内容裁剪与结果缓存 | 先做轻量筛选，再做高成本总结；周报优先复用日摘要；避免对全部原文直接跑重模型 |

### 7.2 结构约束

| 约束 | 影响 |
| --- | --- |
| 当前是 Bun 单仓库原型 | 规格描述保持实现无关，但不假设已有复杂多端系统 |
| 当前暂无独立前端 | 感知渠道以邮件、报告、配置入口、链接与反馈入口为主，不绑定 Web UI |
| MVP 只做 RSS | 所有 MVP 验收只针对 RSS；其他来源只进入后续路线图 |
| 数据存储、部署、模型供应商待定 | 文档定义行为与约束，不绑定具体数据库、云服务或模型厂商 |

### 7.3 Non-Affordances

以下行动被显式排除，不应被产品环境“误暗示”为当前可做：

1. 未经用户授权自动发现、添加或抓取外部来源
2. 直接上传、存储或处理音频/视频原文件
3. 输出无法回链原文的总结、观点或主题判断
4. 在 MVP 中承诺 YouTube、Podcast、Newsletter、微信公众号的完整接入
5. 假设存在复杂前端工作台、拖拽式后台或重型可视化交互
6. 把用户反馈当作只展示不生效的装饰性按钮

## 8. 感知通道与反馈机制

### 8.1 人类 Agent 的感知通道

| 场景 | 用户感知什么 | 系统必须提供什么 |
| --- | --- | --- |
| 添加来源 | 可添加单条 RSS 或批量导入 | 明确的输入入口与导入结果反馈 |
| 阅读 digest | 主题分组、摘要、价值判断 | 结构清晰的日报内容 |
| 校验结论 | 来源、原文链接、证据片段 | 完整 traceability 信息 |
| 调整输出 | 有用/没用、屏蔽来源、摘要风格偏好 | 可操作且语义明确的反馈入口 |

### 8.2 AI Agent 的感知通道

| 场景 | AI 感知什么 | 必要结构 |
| --- | --- | --- |
| 分析内容 | 统一内容模型、来源信息、时间信息 | 标准化 `content-item` |
| 决定分析层级 | 内容长度、候选价值、缓存命中情况 | `analysis-record` 状态与策略版本 |
| 生成 digest | 主题、排序信号、反馈偏好 | `feedback-signal` + 编排规则 |
| 保证可追溯 | 来源标识、原始链接、证据片段 | traceability 元数据 |

### 8.3 系统反馈时间层级

| 层级 | 目标时机 | 反馈示例 |
| --- | --- | --- |
| Immediate | 用户动作后立刻可见 | 导入成功、重复来源被跳过、反馈已记录 |
| Progressive | 抓取或分析过程进行中 | 正在抓取、正在分析、正在生成 digest 的状态信息或日志 |
| Completion | 一轮流程完成后 | 日报已生成并投递、下一轮输出已体现反馈变化 |

## 9. 技术实现导向

本节不是具体架构设计，而是为了避免实现阶段偏离产品约束。

### 9.1 流水线分层

系统应围绕以下产品流水线组织：

1. 来源接入
2. 定时抓取
3. 原始数据落地
4. 清洗与 Markdown 标准化
5. 去重
6. 基础抽取与价值判断
7. 单篇摘要
8. digest 编排
9. 邮件投递
10. 反馈记录与下一轮规则生效

### 9.2 成本控制原则

1. 优先轻模型做筛选与基础抽取
2. 只对高价值内容做更高成本摘要或深分析
3. 对摘要和分析结果做缓存
4. 周报优先复用日报摘要，不重新全量分析

### 9.3 可追溯性原则

1. digest 中的每一条结论都要能回到原文
2. 原文与加工结果必须独立存放
3. 分析结果必须带版本信息，避免不可解释地漂移

## 10. 验收标准与测试场景

### 10.1 MAS-1 验收

| ID | 场景 | 预期结果 |
| --- | --- | --- |
| MAS1-01 | 用户添加单条 RSS | 新来源被创建为 `source`，状态默认为可参与后续抓取 |
| MAS1-02 | 用户批量导入 RSS 且包含重复来源 | 系统不报错；清晰提示已新增、已存在、已跳过数量 |
| MAS1-03 | 调度周期到达 | 系统仅抓取 `active` RSS 来源的新内容 |
| MAS1-04 | 内容进入标准化流程 | 原始内容与清洗后内容分离保存，不互相覆盖 |
| MAS1-05 | 生成日报 | digest 以主题组织，包含摘要、价值判断、来源、原始链接与证据片段 |
| MAS1-06 | 内容价值不足 | 不进入高成本分析，体现分层处理原则 |
| MAS1-07 | 日报投递 | 默认通过电子邮件完成投递，或保留等价报告产物 |

### 10.2 MAS-2 验收

| ID | 场景 | 预期结果 |
| --- | --- | --- |
| MAS2-01 | 用户对内容标记“没用” | 下一轮 digest 中相似内容排序下降或曝光减少 |
| MAS2-02 | 用户屏蔽来源 | 被屏蔽来源不再进入后续编排与投递 |
| MAS2-03 | 用户降低某类主题偏好 | 相关主题在下一轮 digest 中权重下降 |
| MAS2-04 | 用户声明偏好短摘要或深分析 | 下一轮 digest 表达风格发生对应变化 |
| MAS2-05 | 查询反馈记录 | 可以定位到结构化 `feedback-signal` 与其生效对象 |

### 10.3 MAS-3 路线验收

| ID | 场景 | 预期结果 |
| --- | --- | --- |
| MAS3-01 | 新来源类型进入规划 | 仍复用相同领域对象和 traceability 契约 |
| MAS3-02 | 来源是音视频类 | 只接收文本字幕或转写稿，不接收原始音视频文件 |
| MAS3-03 | 团队评估 MVP 完成度 | 非 RSS 来源不构成当前版本交付阻塞 |

### 10.4 关键一致性检查

1. 六条现实约束全部映射到产品要求。
2. MVP 边界与研究文档一致，只覆盖 RSS。
3. 当前项目状态明确写明“Bun 单仓库原型，暂无独立前端”。
4. 所有用户可见 AI 结论都具备 traceability 信息。
5. 反馈闭环不是可选附属功能，而是核心 affordance。

## 11. 附录：术语与边界

### 11.1 术语

- **Affordance**：产品环境对 agent 提供的行动可能性。
- **MAS**：Minimum Affordance Story，最小可闭环、可产生意义的 affordance 故事。
- **Traceability**：从 digest 结论回到来源、链接、原文片段与分析版本的能力。
- **Digest**：按日或按周编排的信息摘要报告；本项目 MVP 以日报为主。

### 11.2 本文档的默认前提

1. 文档语言为中文。
2. 面向内部产品、工程与 AI agent 协作。
3. 主投递渠道默认电子邮件。
4. 长期愿景会写入，但当前交付只锁定 RSS MVP 与反馈闭环。
5. 如后续需要面向干系人的传统 PRD，可从本文档二次抽取，不在本文件内另行维护传统版结构。
